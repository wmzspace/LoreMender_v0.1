    5. 响应式策略

    决策：双布局（移动 + 桌面）通过 @media (min-width: 1024px) 切换。

    - ≤1023px（移动ile:///平板竖屏）：现有布局/平板竖屏）：现有布局 — 无 SideNav、有 Topbar + 
    BottomNav、单列流式、content-wrap 760/1080px
    - ≥1024px（桌面ile:///平板横屏）：新布局/平板横屏）：新布局 — 有 SideNav、无 
    BottomNav、PageTitleBar、多列网格、split-view、content-wrap 920/1180px

    断点选择 1024px 而非 768px：
    - 768px 平板竖屏仍偏窄，split-view 的 420px 面板 + 场景区会挤
    - 1024px 是 iPad 横屏ile:///小笔记本起点，确保/小笔记本起点，确保 split-view 有足够空间
    - 现有 @media (min-width: 700px) 块（滚动条ile:///字号）保留作为/字号）保留作为"平板优化"，1024px 
    块作为"完整桌面"

    实现：
    - App.tsx 渲染 SideNav 时判断：const isDesktop = useMediaQuery('(min-width: 1024px)')（用一个简单的 useMediaQuery
    hook，或直接在 SideNav 组件内 matchMedia）。为避免引入 hook 复杂度，SideNav 始终渲染，但 CSS 默认 display:none，@media≥1024 
    { display:flex }。同理 .app-shell 的 grid 布局只在 @media≥1024 生效，移动端 .app-shell 退化为单列（只有 .app-main）。
    - 各页的桌面布局类（.split-view / .grid-2 等）都写在 @media≥1024 内，移动端不生效。

    useMediaQuery hook（可选，新增 src/lib/useMediaQuery.ts）：
    export function useMediaQuery(query: string): boolean {
      const [matches, setMatches] = useState(() => window.matchMedia(query).matches);
      useEffect(() => {
        const m = window.matchMedia(query);
        const handler = () => setMatches(m.matches);
        m.addEventListener('change', handler);
        return () => m.removeEventListener('change', handler);
      }, [query]);
      return matches;
    }
    用于：App.tsx 决定是否显示 SideNav（CSS 已处理，但 JS 控制更明确）；MiniGamePage 决定 CELL_SIZE。

    9. 执行注记（实现阶段的简化决策）

    1. useMediaQuery hook 简化：不新增 hook 文件。SideNav 始终由 App 渲染，CSS 控制 display:none/flex 切换。WoodenBox 的
    CELL_SIZE 用 CSS 变量 + clamp 无法满足（它是 JS 常量参与布局计算），改为：const CELL_SIZE = window.innerWidth >= 1024 ? 72 :
     48;（MiniGamePage 内一行判断，不引入 hook）。这样 src/lib/useMediaQuery.ts 不需要新建，新增文件从 3 个减为 2 个。
    2. Topbar vs PageTitleBar 取舍：为减少新组件维护成本，不新增 PageTitleBar 组件，而是给现有 Topbar 加 hideBack?: boolean
    prop。桌面端各页调用时传 hideBack（返回由 SideNav 承担），移动端不传保持原样。这样新增文件从 2 个减为 1 个（仅
